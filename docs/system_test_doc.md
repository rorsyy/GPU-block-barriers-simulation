# 系统测试文档

## 1 测试概述

### 1.1 测试目的

本文档旨在从最终用户的操作视角出发，对 GPU 栅栏同步仿真系统进行系统级功能验证。测试关注的是用户在 Web 界面上执行配置、控制、观测等操作后，系统是否给出了正确、及时且符合预期的反馈，而非底层代码的内部逻辑。

### 1.2 测试方法

本文档采用**黑盒测试**方法。测试人员无需了解系统内部实现，仅通过浏览器界面与系统进行交互，依据界面上可观测到的状态变化来判定测试是否通过。

### 1.3 测试环境

如表 1 所示，系统测试在本地单机环境下进行，所有组件均运行于同一台计算机上。

**表 1：测试环境配置**

| 项目        | 配置                            |
| ----------- | ------------------------------- |
| 操作系统    | Windows 10/11                   |
| Python 版本 | 3.7 及以上                      |
| 浏览器      | Chrome / Edge（现代浏览器）     |
| 启动方式    | 命令行执行 `python src/main.py` |
| 访问地址    | `http://localhost:8000`         |

### 1.4 测试用例编号规则

每条测试用例以 "ST-XX" 格式编号（ST 表示 System Test），并在正文中通过编号引用对应的测试设计表（表一）和测试结果表（表二）。

### 2.3 仿真暂停与恢复

本节验证用户在仿真运行过程中点击暂停按钮后，系统能否立即冻结当前状态，并在再次点击启动后从断点处继续推进，而非从头开始。该测试对应 UC-4 中的"暂停"分支。

**用例设计背景**：
- **为什么有这个用例（目的）**：在教学或分析场景中，用户需要频繁冻结画面以观察特定的同步瞬间。此用例验证系统暂停和断点恢复的可靠性，确保暂停时不发生状态泄露，恢复时不会导致时钟重置。

### 2.8 故障注入与容灾行为观察

本节验证用户在故障模式下运行仿真时，系统能否正确模拟线程块故障，并在界面上以红色标记失效的线程块。该测试对应 UC-9（故障注入测试）。

**用例设计背景**：
- **为什么有这个用例（目的）**：真实 GPU 环境中随时可能发生死锁或硬件故障。此用例验证系统的容灾模拟（失联模式）与超时检测机制，确保系统在异常情况下不崩溃并能提供有效反馈。
- **为什么用例是这样（设计逻辑）**：故障注入概率是随机的，因此用例要求“等待约 3～5 轮同步周期”以增加故障触发概率，随后检查是否有节点变红（触发失联）。这是在不修改底层代码注入确定性故障的前提下，验证随机容灾机制最合理的黑盒测试方法。

## 3 栅栏算法完整仿真流程测试

本节针对系统已实现的三种栅栏算法（集中式、树形、静态树形），逐一执行完整的仿真模拟流程测试。每个用例从参数配置、系统初始化开始，经历仿真启动、线程块调度执行、线程块到达栅栏阻塞、栅栏同步释放的全部阶段，直至所有线程块完成至少一轮完整的同步周期。通过按算法逻辑时序逐步观测界面上的状态变化，验证每种栅栏算法的仿真逻辑与仿真结果的正确性和可靠性。

### 3.1 集中式栅栏完整仿真流程

**用例设计背景**：

- **为什么有这个用例（目的）**：集中式栅栏是最基础的同步算法，所有线程块共享同一个全局计数器。本用例旨在验证系统能否正确模拟集中式栅栏的核心行为——线程块逐一到达并签到、计数器累加至总数后触发释放、所有等待中的线程块恢复执行。作为最简单的算法，若集中式栅栏的仿真流程存在错误，则整个仿真引擎的可信度将受到根本性质疑。
- **为什么用例是这样（设计逻辑）**：用例选择 4 个线程块以保持观测的简洁性，采用单步执行模式逐 Tick 推进，便于精确观测每个时钟周期内线程块的状态变迁。测试步骤按照“初始化→启动运行→观测到达过程→观测释放过程→验证恢复执行”的时序排列，完整覆盖了集中式栅栏从开始到结束的每一个关键阶段。由于集中式栅栏不涉及树形拓扑结构，测试中还需确认拓扑面板正确隐藏。
