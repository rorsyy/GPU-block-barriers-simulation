# 概要设计：Model 层关键接口设计与介绍

## 1. 接口设计原则与架构概述

在本仿真系统的设计中，模型层（Model Layer）是包含核心业务逻辑与物理硬件抽象的关键层级。为保证仿真系统的高内聚、低耦合特性，Model 层严格遵循**事件驱动架构（Event-Driven Architecture, EDA）**模式。

整个模型层内部不包含硬编码的直接函数调用依赖，而是主要围绕三个核心实体展开交互：**仿真调度器（Scheduler）**、**执行单元（Block）**和**抽象同步栅栏（Barrier）**。这三者通过**事件总线（EventBus）**作为通信中枢，实现了“控制流”与“数据流”的物理分离。

本文以单次仿真时钟周期（Tick）推进为主线，梳理出整个 Model 层运作流转最关键的三组核心交互接口。这些接口完整闭环了在 GPU 仿真内核中计算单元与硬件同步原语在时序层面的微观协作机制。

---

## 2. 核心交互接口定义与设计说明

按照一个完整的同步周期中数据流动的先后顺序，Model 层内部的接口可以分成三组来理解：第一组是“**推进时间**”——系统时钟滴答一下，所有线程块往前走一步；第二组是“**到达汇报**”——某个线程块做完了当前阶段的任务，向系统报告“我到栅栏了”；第三组是“**释放唤醒**”——所有人都到齐了，栅栏打开，等待中的线程块被重新激活。以下分别详细说明。

### 2.1 仿真时钟推进接口组（自上而下调用）

这组接口是整个仿真的“心跳”，它完成的事情很简单：系统时钟每滴答一下（+1 Tick），调度器就告诉每个正在运行的线程块：“时间过去了一个周期，你们各自向前走一步。”

* **`ScheduleTickInterface` (SimulationModel → Scheduler)**
  * **接口描述**：系统主循环在每个时钟周期（Tick）调用的入口接口，负责推动整个仿真的时间前进。
  * **设计目的**：将“推进时间”的任务交给调度器。这样，主模型就不需要关心当前使用的是“正常调度器”还是“故障注入调度器”，只需按固定节奏给调度器发送时钟滴答即可。
  * **输入参数**：
    | 字段名 | 类型 | 必填 | 说明 |
    |-------|------|------|------|
    | tick | int | 是 | 当前的系统时钟周期数值（例如：1, 2, 3...） |
  * **输出参数**：无

* **`BlockRunStepInterface` (Scheduler → Block)**
  * **接口描述**：调度器命令单个线程块（Block）向前执行一步计算的接口。
  * **设计目的**：调度器通过遍历所有处于“运行中（RUNNING）”的线程块，调用此接口来让它们消耗各自的“工作量”。这模拟了 GPU 中的流式多处理器（SM）安排线程块执行指令的过程。
  * **输入参数**：
    | 字段名 | 类型 | 必填 | 说明 |
    |-------|------|------|------|
    | tick | int | 是 | 当前的系统时钟周期数值 |
  * **输出参数**：
    | 字段名 | 类型 | 说明 |
    |-------|------|------|
    | has_progress | bool | 返回 True 表示该线程块在当前周期成功执行了任务；返回 False 表示它没做任何事（比如处于等待或失败状态）。 |
    | work_done | float | 该线程块在当前时钟周期内实际消耗掉的工作量数值。 |

### 2.2 栅栏到达与同步汇聚接口组（事件驱动触发）

当某个线程块的工作量消耗完毕、到达了同步检查点时，它需要向系统报告“我到栅栏了”。这组接口就是处理这个“报到”过程的：线程块通过事件总线发出到达信号，栅栏对象接收到后进行计数，并判断是否所有人都到齐了。这两个环节通过事件总线解耦，线程块完全不需要知道底层用的是哪种栅栏算法。

* **`ArrivalEventPublishInterface` (Block → EventBus)**
  * **接口描述**：当线程块发现自己的工作量清零、到达了同步点时，向系统总线（EventBus）发送“我到了”的广播接口。
  * **设计目的**：为了让线程块和具体的栅栏算法解耦。线程块只需要在总线上大喊一声“我到栅栏了”，而不需要去管底层到底是集中式栅栏还是树形栅栏。这样保证了代码的整洁和独立性。
  * **输入参数**：
    | 字段名 | 类型 | 必填 | 说明 |
    |-------|------|------|------|
    | type | string | 是 | 事件类型，固定为 BARRIER_ARRIVAL（到达栅栏） |
    | tick | int | 是 | 触发该事件时的时钟周期数值 |
    | data | dict | 是 | 核心数据包，包含刚刚到达的线程块ID (block_id) 和目标栅栏ID (barrier_id) |
    | timestamp | float | 是 | 事件生成的物理时间戳，内部用来排序或打日志 |
  * **输出参数**：无

* **`BarrierArriveInterface` (EventBus/Model → Barrier)**
  * **接口描述**：事件总线监听到有线程块到达后，通知具体的栅栏对象（Barrier）处理该线程块的到达逻辑。
  * **设计目的**：驱动各种不同的栅栏算法开始工作。栅栏对象在这个接口内会去修改全局内存（模拟原子加法 `sim_atomic_add`），并检查是否所有人都到齐了。同时还会通知性能监控模块“发生了一次通信”。
  * **输入参数**：
    | 字段名 | 类型 | 必填 | 说明 |
    |-------|------|------|------|
    | block_id | int | 是 | 刚刚到达栅栏的线程块 ID |
    | tick | int | 是 | 当前的系统时钟周期数值 |
  * **输出参数**：
    | 字段名 | 类型 | 说明 |
    |-------|------|------|
    | triggered_release | bool | 返回 True 表示当前块是最后一个到达的，导致栅栏被打破（释放）；返回 False 表示人还没到齐，继续等待。 |

### 2.3 释放唤醒与控制接管接口组（广播网络路由）

当栅栏检查发现所有线程块都已到齐（例如集中式栅栏的计数器达到了总数，或树形栅栏的根节点收集完所有子树信号），就需要“开闸放行”。这组接口就是处理这个“放行”过程的：栅栏对象通过事件总线广播“栅栏已释放”信号，调度器监听到后，把之前在栅栏前挂起等待的那些线程块重新激活，让它们继续执行下一阶段的任务。

* **`EventPublishInterface` [触发事件: BARRIER_RELEASE] (Barrier → EventBus)**
  * **接口描述**：当栅栏判定所有人到齐时，向事件总线广播“栅栏已释放”信号的接口。
  * **设计目的**：当最后一个线程块到达，栅栏内部变量（例如 Sense 位）发生翻转时，代表同步完成。此时栅栏需要对外喊一声“放行了”，让所有挂起等待的线程块准备恢复执行。
  * **输入参数**：
    | 字段名 | 类型 | 必填 | 说明 |
    |-------|------|------|------|
    | type | string | 是 | 事件类型，固定为 BARRIER_RELEASE（栅栏释放） |
    | tick | int | 是 | 触发释放事件时的时钟周期数值 |
    | data | dict | 是 | 核心数据包，包含被释放的栅栏ID (barrier_id) 以及所有被唤醒的线程块ID列表 (released_block_ids) |
    | timestamp | float | 是 | 事件广播的物理时间戳，内部用来排序或打日志 |
  * **输出参数**：无

* **`BarrierReleaseCallbackInterface` (EventBus → Scheduler)**
  * **接口描述**：调度器（Scheduler）监听到“栅栏已释放”信号后触发的回调处理接口。
  * **设计目的**：调度器收到放行信号后，需要把之前处于“等待状态（WAITING）”的那些线程块重新激活为“运行状态（RUNNING）”，为它们分配下一阶段的计算任务，让它们继续往下执行。
  * **输入参数**：
    | 字段名 | 类型 | 必填 | 说明 |
    |-------|------|------|------|
    | type | string | 是 | 事件类型，固定为 BARRIER_RELEASE |
    | tick | int | 是 | 接收到释放事件时的时钟周期数值 |
    | data | dict | 是 | 核心数据包，包含被释放的栅栏ID和需要被恢复运行状态的线程块ID列表 |
  * **输出参数**：无

### 2.4 全局内存原子访问接口组（被动存储介质交互）

当栅栏对象处理线程块到达逻辑时，需要对全局共享变量（如到达计数器 `counter`、相位标志位 `sense` 等）执行不可分割的"读-改-写"操作。在真实 GPU 硬件中，这些操作由 SM 发起、作用在被动 DRAM 上。本组接口严格遵循该硬件语义，将原子操作逻辑（由 Barrier/SM 持有）与裸存储介质（GlobalMemory/DRAM）进行了物理分离。

* **`GlobalMemoryAtomicAccessInterface` (Barrier → GlobalMemory)**
  * **接口描述**：栅栏对象在执行同步判定时，对全局内存发起的原子级读写操作接口。包含原子加法（`sim_atomic_add`）与原子交换（`sim_atomic_exch`）两个核心操作。
  * **设计目的**：将全局内存建模为纯粹的被动存储介质（DRAM），不包含任何业务逻辑或通信打点。所有原子操作的"发起权"归属于 SM（即 Barrier 对象），内存仅响应最基础的 read/write 请求。这种分离保证了：① 硬件语义的准确映射——DRAM 不会主动计算，原子性由调用方保证；② 通信开销打点（`metrics.record_communication`）统一收敛在 Barrier 层，而非散落在内存模块内部。
  * **输入参数**：
    | 字段名 | 类型 | 必填 | 说明 |
    |-------|------|------|------|
    | key | str | 是 | 全局内存中的变量名（如 `"counter"`、`"sense"` 等共享同步变量） |
    | value / new_value | int / Any | 是 | `sim_atomic_add` 中为累加值（默认 +1），`sim_atomic_exch` 中为要写入的新值 |
  * **输出参数**：
    | 字段名 | 类型 | 说明 |
    |-------|------|------|
    | old_value | int / Any | 返回本次操作前该内存地址的旧值，语义等价于 CUDA `atomicAdd` / `atomicExch` 的返回值。调用方通过比对旧值来判定是否触发栅栏释放（如旧值 == limit - 1 说明当前块是最后到达的） |

## 3. 本章小结

本节详细介绍了 GPU 栅栏仿真系统中 Model 层的核心接口设计。通过引入事件总线（EventBus），系统将线程块（Block）的执行流与栅栏（Barrier）的同步逻辑进行了有效剥离。

在上述 4 个接口组、共 7 个关键交互接口的约束下，系统完成了从“时钟推进”、“同步状态上报”到“屏障释放与唤醒”的完整仿真控制闭环。该设计方案不仅有效降低了各个实体模块之间的代码耦合度，也为后续平滑接入其他种类的栅栏同步算法（如树形栅栏、静态树栅栏等）提供了良好的面向对象扩展基础，符合软件工程高内聚、低耦合的设计原则。
