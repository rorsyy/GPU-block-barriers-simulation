## 全局内存模块与栅栏算法模块 —— 详细设计

---

### 1. 全局内存模块详细设计类图

> 在真实 GPU 硬件中，全局内存 (Global Memory) 是位于芯片外部的 DRAM 存储介质，本身不具备计算能力，仅被动响应来自 SM 的读写请求。本系统将全局内存提取为独立的 `GlobalMemory` 类，严格遵循该硬件语义：内部仅持有一个字典作为物理存储池，对外只暴露 `read()`、`write()` 与 `get_snapshot()` 三个纯访存接口，不包含任何业务逻辑或通信打点行为。`Barrier` 基类通过聚合关系引用该模块，实现了存储介质与控制逻辑的解耦。

```plantuml

skinparam dpi 150

skinparam classFontSize 11
```

### 2. 栅栏算法模块详细设计类图

> 栅栏算法模块是系统仿真引擎的核心，负责模拟 GPU 中线程块到达同步点后的等待与释放逻辑。该模块以抽象基类 `Barrier` 为根，派生出三种具体算法实现：集中式 (`CentralizedBarrier`)、动态树形 (`TreeBarrier`) 和静态树形 (`StaticTreeBarrier`)。`Barrier` 基类承担了 SM 控制逻辑的角色，封装了模拟 GPU 硬件的原子指令 (`sim_atomic_add` / `sim_atomic_exch`)，这些指令内部通过调用所持有的 `GlobalMemory` 实例的 `read()` / `write()` 接口完成"读-改-写"操作，并在每次调用时自动触发 `BarrierMetrics` 记录一次通信开销。两种树形算法分别使用内部类 `Node` 和 `StaticNode` 来构建父子级联的树形拓扑。

### 3. 核心接口与实现细节

#### 3.1 GlobalMemory 接口说明

> `GlobalMemory` 模拟 GPU 片外 DRAM 的被动存储行为。内部以 Python 字典 (`Dict[str, Any]`) 作为物理存储池，对外仅暴露三个无副作用的纯访存接口。该类不持有任何指标收集器，不参与任何业务判定。

| 方法 | 参数 | 返回值 | 说明 |
|------|------|--------|------|
| `__init__(initial)` | `initial: Dict = None` | — | 可选传入初始键值对，若为空则创建空白内存 |
| `read(key, default)` | `key: str`, `default: Any = 0` | `Any` | 读取指定键的当前值，不存在时返回 default |
| `write(key, value)` | `key: str`, `value: Any` | `void` | 将指定键设置为给定值 |
| `get_snapshot()` | — | `Dict` | 返回当前全部内存内容的浅拷贝 |

#### 3.2 Barrier 基类原子指令实现细节

> 在真实 GPU 中，原子指令由 SM 内部的专用 ALU（或 L2 缓存控制器）发起，对全局内存执行不可分割的"读-改-写"操作。本系统在 `Barrier` 基类中封装了两条模拟原子指令，它们的内部实现均通过调用 `GlobalMemory` 的 `read()` 与 `write()` 接口完成，并在每次调用后自动通知 `BarrierMetrics` 记录一次总线通信事件。

| 方法 | 对应 CUDA 指令 | 实现逻辑 |
|------|---------------|---------|
| `sim_atomic_add(key, value)` | `atomicAdd(&addr, val)` | ① `old = mem.read(key)` → ② `mem.write(key, old + value)` → ③ `metrics.record_communication(1)` → ④ `return old` |
| `sim_atomic_exch(key, new_value)` | `atomicExch(&addr, val)` | ① `old = mem.read(key)` → ② `mem.write(key, new_value)` → ③ `metrics.record_communication(1)` → ④ `return old` |

#### 3.3 CentralizedBarrier 核心接口

> 集中式栅栏使用一个共享计数器 `counter` 和一个布尔翻转标志 `sense` 实现同步。当所有 Block 到达后，计数器归零、sense 翻转，触发全局释放。

| 方法 | 触发条件 | 核心行为 |
|------|---------|---------|
| `arrive(block_id, tick)` | Scheduler 检测到 Block 到达同步点 | 调用 `sim_atomic_add("counter", 1)` 递增计数器；若 `count >= limit` 则调用 `release_all()` |
| `release_all(tick)` | `arrive()` 内部判定计数器满 | 翻转 `sense`，调用 `sim_atomic_exch("counter", 0)` 和 `sim_atomic_exch("sense", ...)` 重置内存，标记 `released = True` |
| `check_release(block_id, local_sense)` | Scheduler 轮询 Block 释放状态 | 比较 Block 本地 sense 与全局 sense，若不一致则判定已释放 |

#### 3.4 TreeBarrier 核心接口

> 树形栅栏将 Block 分配到叶节点，到达信号沿树自底向上逐层汇聚；当根节点收齐后，sense 翻转信号自顶向下广播释放所有子节点。

| 方法 | 触发条件 | 核心行为 |
|------|---------|---------|
| `arrive(block_id, tick)` | Block 到达同步点 | 从叶节点开始，逐层向父节点递增 `count`；某层满则重置并继续上传，直至根节点触发 `_release_tree()` |
| `_release_tree(tick)` | 根节点 count 达到 limit | 翻转根节点 sense，调用 `_propagate_sense()` 向下广播 |
| `_propagate_sense(node)` | `_release_tree()` 内部调用 | 递归遍历所有子节点，将父节点的 sense 值传播给每个子节点，每次传播记录一次通信 |
| `_update_memory_snapshot()` | 每次 `arrive()` 执行后 | 将所有树节点的 count 和 sense 写入 `GlobalMemory`，供前端热力图读取 |

### 4. 集中式栅栏完整同步周期时序图

> 下图展示了在 `limit = 3`（3 个 Block）的场景下，集中式栅栏从第一个 Block 到达、到全部到达触发释放、再到所有 Block 被唤醒恢复执行的完整生命周期。重点观察：每次 `arrive()` 调用均会通过 `Barrier` 发起一次对 `GlobalMemory` 的原子加操作，而释放阶段则连续发起两次原子交换操作（重置 counter 和翻转 sense）。

```plantuml
@startuml
' === A4 黑白打印优化 ===
scale max 720 width
skinparam dpi 150
skinparam defaultFontSize 10
skinparam sequenceMessageFontSize 9
skinparam sequenceParticipantFontSize 10
skinparam backgroundColor white
skinparam shadowing false
skinparam monochrome true

participant "Scheduler" as SCH
participant "CentralizedBarrier" as BAR
participant "GlobalMemory" as MEM
participant "BarrierMetrics" as MET
participant "Block 0" as B0
participant "Block 1" as B1
participant "Block 2" as B2

== Block 0 到达 ==
SCH -> B0 : set_waiting(tick)
SCH -> BAR : arrive(0, tick)
activate BAR
BAR -> MEM : read("counter")
MEM --> BAR : old=0
BAR -> MEM : write("counter", 1)
BAR -> MET : record_communication(1)
BAR --> SCH : false (未满)
deactivate BAR

== Block 1 到达 ==
SCH -> B1 : set_waiting(tick)
SCH -> BAR : arrive(1, tick)
activate BAR
BAR -> MEM : read("counter")
MEM --> BAR : old=1
BAR -> MEM : write("counter", 2)
BAR -> MET : record_communication(1)
BAR --> SCH : false (未满)
deactivate BAR

== Block 2 到达 (触发释放) ==
SCH -> B2 : set_waiting(tick)
SCH -> BAR : arrive(2, tick)
activate BAR
BAR -> MEM : read("counter")
MEM --> BAR : old=2
BAR -> MEM : write("counter", 3)
BAR -> MET : record_communication(1)
note right : count=3 >= limit=3

BAR -> BAR : release_all(tick)
activate BAR
BAR -> MEM : read("counter") → exch
BAR -> MEM : write("counter", 0)
BAR -> MET : record_communication(1)
BAR -> MEM : read("sense") → exch
BAR -> MEM : write("sense", 1)
BAR -> MET : record_communication(1)
BAR -> MET : record_release(tick)
deactivate BAR

BAR --> SCH : true (已释放)
deactivate BAR

== Scheduler 唤醒所有 Block ==
SCH -> B0 : resume()
SCH -> B1 : resume()
SCH -> B2 : resume()

@enduml
```

### 5. TreeBarrier 树形传播时序图

> 下图展示了在 `limit = 4`（4 个 Block，二叉树 radix=2）的场景下，树形栅栏的完整到达与释放过程。4 个 Block 被分配到两个叶节点（每个叶节点管理 2 个 Block）。到达信号自底向上逐层汇聚：当一个叶节点收齐 2 个 Block 后，向父节点上传；当根节点也收齐后，sense 翻转信号沿树自顶向下广播释放。每一次跨节点的信号传递都会被 `BarrierMetrics` 记录为一次通信开销。

```plantuml
@startuml
' === A4 黑白打印优化 ===
scale max 720 width
skinparam dpi 150
skinparam defaultFontSize 10
skinparam sequenceMessageFontSize 9
skinparam sequenceParticipantFontSize 10
skinparam backgroundColor white
skinparam shadowing false
skinparam monochrome true

participant "Scheduler" as SCH
participant "TreeBarrier" as BAR
participant "Leaf_0\n(管理Block 0,1)" as L0
participant "Leaf_1\n(管理Block 2,3)" as L1
participant "Root" as ROOT
participant "GlobalMemory" as MEM
participant "BarrierMetrics" as MET

== 阶段一：自底向上汇聚 ==

SCH -> BAR : arrive(0, tick)
activate BAR
BAR -> L0 : count++ (0->1)
BAR -> MET : record_communication(1)
note right : Leaf_0: 1/2, 未满
BAR -> MEM : write("node_0.count", 1)
BAR --> SCH : false
deactivate BAR

SCH -> BAR : arrive(1, tick)
activate BAR
BAR -> L0 : count++ (1->2)
note right : Leaf_0: 2/2, 满！重置并上传
BAR -> L0 : count = 0
BAR -> MET : record_communication(1)
BAR -> ROOT : count++ (0->1)
note right : Root: 1/2, 未满
BAR -> MEM : write("node_0.count", 0)
BAR -> MEM : write("node_2.count", 1)
BAR --> SCH : false
deactivate BAR

SCH -> BAR : arrive(2, tick)
activate BAR
BAR -> L1 : count++ (0->1)
BAR -> MET : record_communication(1)
note right : Leaf_1: 1/2, 未满
BAR -> MEM : write("node_1.count", 1)
BAR --> SCH : false
deactivate BAR

SCH -> BAR : arrive(3, tick)
activate BAR
BAR -> L1 : count++ (1->2)
note right : Leaf_1: 2/2, 满！重置并上传
BAR -> L1 : count = 0
BAR -> MET : record_communication(1)
BAR -> ROOT : count++ (1->2)
note right : Root: 2/2, 满！触发释放

== 阶段二：自顶向下广播释放 ==
BAR -> ROOT : sense = !sense
BAR -> MET : record_release(tick)

BAR -> L0 : sense = root.sense
BAR -> MET : record_communication(1)
BAR -> L1 : sense = root.sense
BAR -> MET : record_communication(1)

BAR -> MEM : write("node_2.sense", ...)
BAR -> MEM : write("node_0.sense", ...)
BAR -> MEM : write("node_1.sense", ...)

BAR --> SCH : true (已释放)
deactivate BAR

@enduml
```
