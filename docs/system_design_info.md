

### （二）拟解决的主要问题

1.  **SIMT架构的串行模拟问题**：如何在串行执行的CPU程序中，逼真模拟GPU“单指令多线程”的并发行为[2][5]？
    *   *拟解决方法*：计算机（CPU）本质是串行执行的。拟采用“时间片轮转”或“逻辑时钟步（Tick）”的方式。在每一个逻辑时钟步内，循环更新所有Block的状态，从而在宏观上产生“同时运行”的视觉效果。

2.  **线程块间无法直接同步的问题**：当前 GPU 架构仅提供线程块内部的同步原语（如 `__syncthreads()`），缺乏跨线程块的硬件级同步机制。对于需要全局一致性的并行程序而言，线程块之间无法协调执行进度会导致数据不一致、阶段性任务无法正确推进等问题[2][6]。如何解决在仅依赖软件手段的前提下，实现一种可靠且行为可预测的线程块级栅栏机制，使所有线程块在特定阶段形成一致性？
    *   *拟解决方法*：采用成熟的 sense-reversing barrier 思想[12]，将全局计数变量和“阶段标志”（sense bit）相结合，使得所有线程块能够在每个栅栏周期中通过标志反转来判断同步点的达成。该方法具有逻辑清晰、可扩展性好、在 CPU 并行模型中已有理论基础，可作为 GPU 环境下跨线程块同步的适配方案[3][4]。

3.  **高并发状态下的可视化性能**：当模拟较多线程块且高频更新状态时，如何保障界面刷新流畅，避免UI卡顿。
    *   *拟解决方法*：采用双缓冲绘图技术（Double Buffering）或局部刷新策略，并实现逻辑计算与界面渲染的分离（MVC模式）。后端模拟引擎全速运行，前端界面采用定时采样（如每秒30帧）的方式批量更新UI，并利用双缓冲绘图技术提升渲染效率[10][11]。

4.  **跨线程块同步行为难以观察与教学困难的问题**：GPU 中线程块调度不透明，实际同步行为不可直接观察，给教学与并行编程入门带来障碍。
    *   *拟解决方法*：基于软件模拟的方式，将栅栏内部的等待、计数、反转（sense-reversing）、释放等过程可视化，帮助学习者理解跨线程块同步的运行机理[13][14]。

## 三、研究方法及措施

1.  **面向对象分析与设计（OOAD）**：利用UML工具绘制类图（Class Diagram）和时序图（Sequence Diagram），明确Scheduler对Block的调度逻辑；设计状态机（State Machine），精确定义线程块在Ready、Running、Waiting、Blocked状态间的转换条件[10]。
2.  **软件工程方法（MVC设计模式）**：
    *   **Model（模型层）**：封装栅栏同步的核心模拟逻辑，包括线程块状态管理、栅栏控制及同步异常检测等功能。
    *   **View（视图层）**：负责 Inter-Block 栅栏同步过程的可视化展示，以及用户操作指令的输入采集。
    *   **Controller（控制层）**：负责接收用户操作、驱动仿真流程推进，并协调表现层与业务逻辑层之间的信息交互[10][11]。

## 四、系统设计

本系统面向 GPU 并行计算中线程合作同步的研究与教学需求，基于已有 Inter-Block 栅栏同步相关研究工作，设计并实现一个线程块级栅栏同步模拟软件。系统以线程块（Block）作为最小同步参与实体，通过抽象建模的方式，对多个线程块在执行过程中的跨块协作与等待行为进行模拟，并以图形化方式直观展示 Inter-Block 栅栏机制的工作过程。

Inter-Block 同步需要在线程块之间建立全局协作关系，其实现通常依赖于软件层面的栅栏协议与状态管理机制。本系统不关注线程块内部的线程或线程束执行细节，而是将线程块视为整体执行单元，从宏观层面刻画线程块级合作同步。

### （一）系统功能需求分析

结合课题目标与 Inter-Block 同步机制的特点，系统需满足以下功能需求：

1.  **Inter-Block 栅栏同步行为模拟功能**：系统应能够模拟多个线程块在执行过程中通过栅栏机制进行跨块同步的过程，准确反映线程块到达栅栏、等待以及同步释放等阶段性状态变化。
2.  **线程块执行状态的可视化展示功能**：系统应以线程块为基本展示对象，通过图形化方式呈现各线程块在运行、等待栅栏及同步完成等状态下的变化情况，从整体上展示跨线程块的协同执行关系。
3.  **仿真过程的交互式控制功能**：系统应支持用户对仿真过程进行控制，包括启动、暂停及单步推进等操作，以便逐步观察 Inter-Block 同步过程的动态演化。
4.  **关键同步参数的配置功能**：系统应允许用户设置参与同步的线程块数量、栅栏触发条件等参数，以分析不同配置下 Inter-Block 栅栏同步行为的差异。
5.  **跨线程块同步异常检测功能**：系统应具备对 Inter-Block 同步异常的检测能力。当部分线程块无法到达栅栏，导致同步无法完成时，系统能够识别并提示该异常状态，辅助分析同步问题。

### （二）系统总体架构设计

为满足上述功能需求，系统采用分层架构设计，并结合 MVC（Model–View–Controller）设计模式，实现系统功能与结构的清晰解耦。
在逻辑结构上，系统划分为以下三层：
1.  **表现层（View Layer）**：负责 Inter-Block 栅栏同步过程的可视化展示，以及用户操作指令的输入采集。
2.  **控制层（Controller Layer）**：负责接收用户操作、驱动仿真流程推进，并协调表现层与业务逻辑层之间的信息交互。
3.  **业务逻辑层（Model Layer）**：封装 Inter-Block 栅栏同步的核心模拟逻辑，包括线程块状态管理、栅栏控制及同步异常检测等功能。

### （三）系统模块划分

在总体架构基础上，系统进一步划分为若干核心功能模块，各模块按照职责分布在不同架构层中。

1.  **可视化交互模块（表现层）**：该模块负责以图形化方式展示多个线程块在 Inter-Block 栅栏同步过程中的状态变化。每个可视化单元对应一个线程块，不同颜色或标记表示其当前执行状态，如运行中、等待栅栏或同步完成。模块同时提供仿真控制与参数配置的图形界面。
2.  **仿真控制与协调模块（控制层）**：该模块负责监听用户操作事件，并将其转换为仿真控制指令；同时维护仿真的执行节奏，通过单步或周期性推进的方式驱动业务逻辑层更新线程块状态。
3.  **线程块结构与状态管理模块（业务逻辑层）**：该模块用于抽象线程块这一执行实体，并维护其运行状态信息，包括线程块是否已到达栅栏、是否处于等待状态以及是否已完成同步等，为 Inter-Block 栅栏控制提供基础数据支持。
4.  **Inter-Block 栅栏同步核心模块（业务逻辑层）**：该模块是系统的核心，用于模拟跨线程块的栅栏同步机制。模块维护全局栅栏状态，当线程块执行至同步点时，将其标记为已到达栅栏；当所有参与同步的线程块均到达栅栏后，栅栏被触发释放，所有线程块同步进入下一执行阶段，从而实现线程块级合作同步。
5.  **同步异常诊断模块（业务逻辑层）**：该模块用于检测 Inter-Block 同步过程中可能出现的异常情况。例如，当部分线程块长期无法到达栅栏，导致同步无法完成时，系统将识别该状态并向用户提示潜在的同步异常，为分析和理解 Inter-Block 同步问题提供支持。
