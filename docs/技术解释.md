一、底层架构核心决策：逻辑时钟步（Tick-based）的离散事件仿真（DES）
在 GPU 栅栏模拟系统中，采用 逻辑时钟步（Tick-based）的离散事件仿真（Discrete Event Simulation, DES） 而不是原生的多线程/协程并发，是一个极为关键的底层架构决策。以下是采用离散事件仿真的核心原因及其必要性的详细解释：
1. 完美映射真实硬件的时钟周期模型
•必要性与真实性：真实的 GPU 硬件（流式多处理器 SM）是严格依靠晶振产生的高频硬件时钟周期（Clock Cycles）来调度指令和控制微架构状态的。
•优势：离散事件仿真中的一个逻辑“Tick”（嘀嗒），在代码层面完美映射了 GPU 的一个物理时钟或执行周期。这种基于时间步（Time-stepped）的推进方式，使得模拟器在宏观上能以极高逼真度还原 GPU 指令发射、内存读写耗时和同步等待的微观时序特征。
2. 规避原生多线程并发在模拟系统中的“薛定谔陷阱”
•必要性：如果系统使用 Python 原生的多线程（threading）或协程（asyncio）去直接模拟 64 个甚至成百上千个 GPU 线程块，将面临致命灾难。操作系统层面的线程调度（受限于 CPU 负载和 Python GIL 全局解释器锁）充满了不可预测性。
•优势：采用单线程驱动的离散Tick仿真引擎，彻底消除了底层操作系统调度带来的干扰。一切“并发”在上帝视角下都是以 Tick 为最小单位严格受控的。这意味着相同的配置和随机种子，能够跑出100% 确定且可复现的仿真结果，这对于排查极难复现的“死锁（Deadlock）”或竞态条件等算法Bug有着不可替代的价值。
3. 实现绝对公平的微秒级量化对比机制
•必要性：这是一款用于研究和对比栅栏算法（如 Centralized 对比 Tree）性能差距的软件，其核心评价指标是同步延迟（Latency）。如果依赖系统真实物理时间（如 time.time()），主机 CPU 突然的卡顿就会导致数据彻底失效。
•优势：在 DES 机制下，“时间”被抽象成了虚拟的 Tick 计数器。不论物理主机执行一个 Tick 花了 1 毫秒还是 1 秒，模型世界里经历的时间永远是 1 个单位。这为计算不同算法的核心性能指标提供了绝对公平、严谨的客观参考标尺，使得最终产出的对比图表（Charts）具有极高的科学研究和论文论证价值。
4. 天生契合可视化系统的“上帝视角”快照需求
•必要性：前端可视化界面需要展示全局内存热力图、所有 Block 的实时运行进度。这就要求后端必须定期冻结整个世界，提供一份绝对一致（Consistent）的状态快照（Snapshot）。
•优势：针对原生多线程去抓取一致性快照需要极其复杂的全局锁机制，而离散事件仿真使得这一切变成自带功能。在每一个 Tick 结束时，整个系统的状态天然静止且逻辑闭环。系统可以通过 PAUSE 或 STEP（单步调试）指令随时冻结时间轴，将当前 Tick 的快照无缝转化为 JSON 提供给前端展示，实现了极佳的交互与教学演示体验。
二、MVC 分层架构
选用原因与必要性
GPU 同步仿真系统内部包含大量高复杂度计算逻辑，例如时钟周期推进、多线程块并行行为模拟、同步状态更新等。同时，前端可视化界面需要高频刷新 Canvas 或 DOM 节点以实现流畅动画。若将仿真计算逻辑与界面渲染逻辑耦合在同一模块中，会造成严重的执行阻塞、界面卡顿以及代码结构混乱，难以维护与扩展。
技术优势
MVC 架构通过关注点分离原则，将系统划分为模型层、视图层与控制层。其中，模型层对应后端仿真计算引擎，视图层负责前端数据展示与渲染，控制层统一管理交互逻辑与生命周期调度。各层之间通过标准化接口通信，使后端 Python 仿真引擎可专注于逻辑时钟推进与状态计算，前端 JavaScript 则以独立帧率异步拉取仿真快照并完成绘制，实现计算与渲染互不干扰，显著提升系统整体运行效率与稳定性。
三、EventBus 事件总线与发布-订阅模式
选用原因与必要性
在 GPU 硬件行为模拟中，线程块与栅栏同步指令之间存在复杂的异步交互关系：线程块到达栅栏后需挂起等待，满足同步条件后再被唤醒执行。在传统紧耦合设计中，调度器与栅栏对象相互直接调用、互相修改内部状态，极易引发状态不一致、数据竞争甚至逻辑死锁。
技术优势
引入事件总线与发布-订阅模式后，系统实现了组件间完全解耦。调度器检测到线程块抵达栅栏时，仅需发布 BARRIER_ARRIVAL 事件，无需关心具体处理对象；栅栏模块监听对应事件并完成计数，当满足释放条件时发布 BARRIER_RELEASE 事件。这种异步、非阻塞的通信方式避免了组件间强依赖，后续新增异常场景（如线程超时、模拟断开、强制唤醒等）时，无需修改核心交互结构，系统可扩展性与鲁棒性显著提升。
四、调度器模块中的策略模式
选用原因与必要性
本系统定位于教学演示与科研仿真工具，需要支持多种运行环境以满足不同实验需求，包括常规随机调度模式、可复现的公平对比模式、以及用于容错测试的异常注入模式。若将多种调度逻辑硬编码在主循环中，会导致代码臃肿、分支复杂且难以维护。
技术优势
策略模式将不同调度规则抽象为独立策略类，如 NormalScheduler、ComparisonScheduler、FailureScheduler 等，统一对外提供一致接口。控制层根据前端配置在运行时动态加载并切换调度策略，主仿真引擎仅通过统一的 schedule_tick() 方法驱动执行，无需感知底层策略差异。该设计严格遵循开闭原则，在不修改核心框架的前提下支持新增调度策略，便于实验扩展与对比测试。
五、栅栏算法模块中的工厂模式
选用原因与必要性
系统需支持集中式、树形、蝶形等至少七种栅栏同步算法，不同算法在初始化结构、数据组织方式与内存布局上差异巨大。例如集中式栅栏仅需单个共享计数器，而树形栅栏与蝶形栅栏需要递归构建多层节点结构并建立层级映射关系。若在主流程中直接创建各类栅栏对象，会导致初始化逻辑高度复杂、耦合严重。
技术优势
通过工厂模式封装栅栏对象的创建过程，对外提供统一的工厂接口 BarrierFactory。仿真核心只需传入线程块数量、算法类型等参数，工厂即可自动完成复杂结构构建并返回符合统一抽象接口的栅栏实例，接口包含 arrive、check_release 等标准方法。该方式屏蔽了底层对象构造细节，简化核心控制流程，保证多算法兼容，同时便于后续新增栅栏同步机制。
