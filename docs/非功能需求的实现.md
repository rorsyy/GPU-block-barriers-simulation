针对您刚刚提出的问题：“**这些非功能需求都是怎样实现的**”，以下是结合系统现有代码架构的具体实现原理解析：

### 3.5.1 性能需求 (Performance)

1. **仿真步进频率 (≥ 20 Ticks/sec)**
   *   **实现机制**：系统在后端 `SimulationController` 中开辟了一个独立的后台守护线程（`self.simulation_thread`）。当状态为 `RUNNING` 时，该线程会在一个 `while` 循环中不断调用 `self.model.schedule_tick()` 来推进核心仿真逻辑，紧接着调用 `time.sleep(0.05)`。这个 `0.05` 秒的硬编码延时，严格保证了主循环以 $1 \div 0.05 = 20$ 次/秒 的稳定频率平滑推进，不会因为死循环而占满单核 CPU，也给前端留出了极其舒适的动画渲染冗余时间。
2. **通信接口响应 (≤ 100ms)**
   *   **实现机制**：前端对 `/api/state` 的轮询极其高频。为了保证毫秒级响应，后端 `http.server` 在处理该接口时**没有任何磁盘 I/O 或数据库查询**。所有的仿真状态（Block 状态、内存键值对、统计指标）完全保存在内存字典中（`dict`）。获取快照（`get_state_snapshot`）仅仅是执行了一次内存层面的浅拷贝和 JSON 序列化（`json.dumps`），时间复杂度极低，响应时间通常在 2~5ms 级别，远低于 100ms 的红线。

### 3.5.2 可扩展性需求 (Extensibility)

1. **算法拓展松耦合**
   *   **实现机制**：系统采用了经典的**策略模式 (Strategy Pattern)** 与**面向接口编程**。在 `src/model/barrier.py` 中，定义了 `Barrier` 抽象基类（继承自 `ABC`）。前端传来的字符串（如 `"Tree"`）会通过工厂方法动态实例化对应的子类对象。任何未来的新算法，只需新建一个类继承 `Barrier`，按照契约实现底层的 `arrive()` 和 `check_release()` 等纯逻辑即可，完全不需要改动 HTTP 路由、Web 界面渲染逻辑或是时钟推进器。
2. **事件总线可扩展**
   *   **实现机制**：利用了**发布-订阅模式 (Pub-Sub)**。在 `src/model/event_bus.py` 中实现了一个轻量级的事件中心。调度器（Scheduler）在推动线程块运行遇到栅栏时，并不直接修改界面，而是向 EventBus 发送一条 `BARRIER_ARRIVAL` 事件；性能指标收集器（`BarrierMetrics`）只需在初始化时通过 `event_bus.subscribe()` 订阅该事件，就能无侵入地完成通信次数的统计。新增监控模块只需“挂载”监听即可，实现了真正的模块间解耦。

### 3.5.3 跨平台需求 (Crossplatform)

1. **环境零依赖与一键部署**
   *   **实现机制**：后端代码严格贯彻了**零第三方库依赖**原则。无论是 HTTP Web 服务（使用了内置的 `http.server`）、并发控制（使用了内置的 `threading`）、还是数据序列化（使用了内置的 `json`），全都只使用 Python 3.7+ 的标准库（Standard Library）。这导致整个项目没有 `requirements.txt` 需要通过 pip 安装，用户只要装了 Python，在任何平台拉下代码直接运行 `python main.py` 就能跑通。
2. **全平台浏览器兼容**
   *   **实现机制**：前端完全抛弃了 Vue、React 等重型框架，采用了**原生三剑客**（Vanilla JS + 原生 HTML5 + 原生 CSS3 变量与 Flex/Grid 布局）。由于没有使用任何超前的、带有实验性质的 Web API（如 WebGPU），且 DOM 更新采用了基础的 `getElementById` 和 `innerHTML` / `className` 操作，使得这套 UI 能够毫无兼容性负担地跑在 Windows、macOS 和 Linux 下的任何主流现代浏览器（Chrome, Edge, Firefox, Safari）上。
