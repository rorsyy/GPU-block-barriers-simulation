        GPU栅栏仿真系统 —— 硬件与软件对应关系详细说明

一、硬件对应关系总览
================================================================================

本项目的仿真模型在软件层面模拟了GPU硬件中栅栏同步所依赖的关键硬件机制。
下面从四个维度详细说明：原子操作、内存层次、线程块通信、树形结构。

────────────────────────────────────────────────────────────────────────────────
1.2 原子操作 —— 真实硬件机制与软件模拟
────────────────────────────────────────────────────────────────────────────────

【真实GPU硬件中的原子操作】

原子操作是GPU硬件提供的一种特殊内存访问指令，在硬件层面保证"读-改-写"
操作的不可分割性(Atomicity)。

硬件实现原理 (以NVIDIA GPU为例):
  1) SM发出原子操作请求，经过L1 Cache到达L2 Cache
  2) L2 Cache中有专用的原子操作单元(Atomic Unit, AU)
  3) AU对目标内存地址加锁，执行"读旧值→计算→写新值"三步操作
  4) 操作完成后解锁，返回旧值给发起请求的线程
  5) 如果多个SM同时对同一地址发起原子操作，硬件会串行化处理

  ┌────────┐    ┌──────────┐    ┌──────────────────────────┐
  │ SM     │───>│ L1 Cache │───>│ L2 Cache                 │
  │(线程块)│    │ (SM私有) │    │ ┌──────────────────────┐ │
  │        │    │          │    │ │ Atomic Unit (AU)     │ │
  │        │    │          │    │ │                      │ │
  │        │    │          │    │ │ 1. Lock(addr)        │ │
  │        │    │          │    │ │ 2. old = Read(addr)  │ │
  │        │    │          │    │ │ 3. Write(addr, new)  │ │
  │        │    │          │    │ │ 4. Unlock(addr)      │ │
  │        │    │          │    │ │ 5. Return old        │ │
  │        │<───│          │<───│ └──────────────────────┘ │
  │(收到old)│   │          │    │                          │
  └────────┘    └──────────┘    └──────────────────────────┘

常用原子指令:
  atomicAdd(addr, val)   - 原子加法,  返回旧值
  atomicSub(addr, val)   - 原子减法,  返回旧值
  atomicExch(addr, val)  - 原子交换,  返回旧值
  atomicMin(addr, val)   - 原子取最小,返回旧值
  atomicMax(addr, val)   - 原子取最大,返回旧值
  atomicCAS(addr, cmp, val) - 比较并交换(CAS), 返回旧值

竞争问题:
  当多个线程块同时对同一全局内存地址执行原子操作时，会发生竞争(Contention)。
  硬件的AU必须将这些并发请求串行化，导致性能下降。
  这就是 Centralized Barrier 在大规模场景下性能差的根本原因——
  所有Block竞争同一个counter变量。

【软件模拟对应】

  本项目在 barrier.py 基类中用 Python 方法模拟原子指令:

  ┌──────────────────────────┬───────────────────────────────────────┐
  │     软件模拟方法          │     对应的真实CUDA原子指令             │
  ├──────────────────────────┼───────────────────────────────────────┤
  │ sim_atomic_add(key, val) │ atomicAdd(&global_memory[key], val)   │
  │                          │ 读取旧值→加val→写回→返回旧值         │
  ├──────────────────────────┼───────────────────────────────────────┤
  │ sim_atomic_exch(key, val)│ atomicExch(&global_memory[key], val)  │
  │                          │ 读取旧值→写入新值→返回旧值           │
  └──────────────────────────┴───────────────────────────────────────┘

  虽然Python本身是单线程串行的（不存在真正的并发竞争），但在逻辑模型上：
  - 每次调用都被视为一次原子的"读-改-写"操作
  - 每次调用都通过 metrics.record_communication(1) 记录通信开销
  - global_memory 字典的键名对应全局内存中的变量地址

────────────────────────────────────────────────────────────────────────────────
1.3 内存读取/写入 —— Sense翻转轮询机制
────────────────────────────────────────────────────────────────────────────────

【真实硬件中的Sense轮询】

在真实GPU中，栅栏释放信号通过"sense翻转"机制传播：
  - 每个线程块维护一个本地的 local_sense 变量（存在寄存器中）
  - 全局内存中有一个共享的 global_sense 变量
  - 等待中的线程块不断读取 global_sense 并与 local_sense 比较

  // 真实CUDA伪代码
  __device__ void wait_for_release(volatile int* global_sense, int local_sense) {
      while (*global_sense == local_sense) {
          // busy-wait (自旋等待)
          // 每次循环都是一次全局内存读取
      }
      // global_sense 翻转了 → 栅栏已释放
  }

  注意 volatile 关键字：
  告诉编译器不要将 global_sense 缓存到寄存器中，
  每次循环必须从全局内存重新读取，否则可能永远看不到更新。

【软件模拟对应】

  本项目通过 Block.update_sense() 方法模拟轮询过程:
    - 每个Tick调用一次 check_release(block_id, local_sense)
    - 如果 local_sense != global_sense → 释放信号到达，恢复执行
    - 每次 check_release 记录一次通信开销（对应一次全局内存读取）

二、线程块怎样调用原子操作 —— 详细硬件过程
================================================================================

────────────────────────────────────────────────────────────────────────────────
2.1 硬件层面的原子操作调用流程
────────────────────────────────────────────────────────────────────────────────

一个线程块(Block)中的某个线程执行原子操作时，硬件经历以下步骤：

步骤1: 指令发射
  SM中的Warp调度器(Warp Scheduler)选中包含原子指令的Warp，
  将原子操作指令发射到LD/ST单元(Load/Store Unit)。

步骤2: 地址计算
  LD/ST单元计算目标全局内存地址（如 &counter 的物理地址）。

步骤3: L1 Cache Bypass
  原子操作通常绕过L1 Cache（因为L1是SM私有的，无法保证跨SM一致性），
  直接发送到L2 Cache。

步骤4: L2 Atomic Unit执行
  L2 Cache中的原子操作单元(AU)：
    a) 锁定目标缓存行(Cache Line, 128 bytes)
    b) 从DRAM加载数据（如果不在L2中）
    c) 读取旧值 old_value
    d) 计算新值 new_value = old_value + increment
    e) 写回新值到缓存行
    f) 解锁缓存行
    g) 将 old_value 返回给发起请求的SM

步骤5: 结果返回
  SM收到返回的旧值，存入目标寄存器，线程继续执行后续指令。

  时间线：
  ┌────────┐    ┌────────┐    ┌──────────┐    ┌────────┐
  │ 指令   │───>│ 地址   │───>│ L2 AU    │───>│ 结果   │
  │ 发射   │    │ 计算   │    │ 原子执行 │    │ 返回   │
  │ ~1cyc  │    │ ~1cyc  │    │ ~100cyc  │    │ ~100cyc│
  └────────┘    └────────┘    └──────────┘    └────────┘
                               ↑ 如果竞争严重
                               │ 可能需要排队等待
                               │ 延迟可达数百cycle

────────────────────────────────────────────────────────────────────────────────
2.2 软件模拟中的调用示例
────────────────────────────────────────────────────────────────────────────────

以 CentralizedBarrier 为例，Block到达栅栏时的原子操作序列:

  ┌─────────────────────────────────────────────────────────────────┐
  │  Block_i 到达栅栏 → arrive(block_id, tick)                     │
  │                                                                 │
  │  第1步: 原子加法                                                │
  │    old_count = sim_atomic_add("counter", 1)                     │
  │    ≡ CUDA: old = atomicAdd(&counter, 1);                        │
  │    → global_memory["counter"] += 1                              │
  │    → 记录1次通信                                                │
  │                                                                 │
  │  第2步: 判断是否最后到达                                        │
  │    if (old_count + 1 >= limit):                                 │
  │      // 是最后一个Block → 执行释放                              │
  │                                                                 │
  │      第3步: 原子重置计数器                                      │
  │        sim_atomic_exch("counter", 0)                            │
  │        ≡ CUDA: atomicExch(&counter, 0);                         │
  │        → 记录1次通信                                            │
  │                                                                 │
  │      第4步: 原子翻转sense                                       │
  │        sim_atomic_exch("sense", new_sense)                      │
  │        ≡ CUDA: atomicExch(&sense, !sense);                      │
  │        → 记录1次通信                                            │
  │                                                                 │
  │  总计: 非最后到达Block = 1次通信                                │
  │        最后到达的Block = 3次通信                                │
  └─────────────────────────────────────────────────────────────────┘

三、线程块如何进行通信 —— 硬件过程
================================================================================

────────────────────────────────────────────────────────────────────────────────
3.1 核心约束：线程块间只能通过全局内存通信
────────────────────────────────────────────────────────────────────────────────

GPU硬件架构决定了线程块间通信的唯一通道：

  Block_i (SM_x)                              Block_j (SM_y)
  ┌──────────────┐                            ┌──────────────┐
  │ Registers    │  ← 不可跨Block访问         │ Registers    │
  │ Shared Mem   │  ← 不可跨Block访问         │ Shared Mem   │
  │ L1 Cache     │  ← SM私有，不保证一致性     │ L1 Cache     │
  └──────┬───────┘                            └──────┬───────┘
         │                                           │
         │    ┌──────────────────────────────┐       │
         └───>│       L2 Cache (共享)        │<──────┘
              │  ┌────────────────────────┐  │
              │  │ 原子操作单元(AU)       │  │
              │  │ 保证并发写入的正确性   │  │
              │  └────────────────────────┘  │
              └──────────────┬───────────────┘
                             │
              ┌──────────────┴───────────────┐
              │     Global Memory (HBM/DRAM) │
              │                              │
              │  counter = N                 │  ← 栅栏计数器
              │  sense   = 0/1              │  ← 释放信号
              │  node[0].count = ...        │  ← 树节点状态
              │  node[0].sense = ...        │
              └──────────────────────────────┘

为什么不能用共享内存(Shared Memory)跨Block通信？
  - 共享内存只在同一个SM的同一个Block内可见
  - 不同Block可能调度到不同SM上执行，甚至同一SM上分时执行
  - 硬件没有提供跨SM的共享内存互联

────────────────────────────────────────────────────────────────────────────────
3.2 通信模式详解
────────────────────────────────────────────────────────────────────────────────

模式A: 原子写入 + 轮询读取 (Centralized Barrier)

  Block_0:   atomicAdd(&counter, 1)  →  写入全局内存  (1次通信)
  Block_1:   atomicAdd(&counter, 1)  →  写入全局内存  (1次通信)
  Block_2:   atomicAdd(&counter, 1)  →  写入全局内存  (1次通信)
  ...
  Block_N-1: atomicAdd(&counter, 1)  →  写入全局内存  (1次通信)
             counter == N → atomicExch(&sense, !sense)  (1次通信)

  所有等待中的Block:
    while(load(&sense) == local_sense) {}  ← 每次循环1次全局内存读取
    // sense翻转后退出循环 → 栅栏释放

  特点: 所有Block竞争同一地址，高竞争(High Contention)

模式B: 树形聚合 + 广播 (Tree/Static Tree Barrier)

  到达阶段(聚合，自底向上):
    Block_0 → atomicAdd(&node_leaf.count, 1)  → 叶子节点  (1次通信)
    Block_1 → atomicAdd(&node_leaf.count, 1)  → 叶子节点  (1次通信)
    叶子满 → 向父节点传播: atomicAdd(&node_parent.count, 1)  (1次通信)
    逐层向上直到根节点

  释放阶段(广播，自顶向下):
    root.sense = !root.sense
    → 写入 child_0.sense  (1次通信)
    → 写入 child_1.sense  (1次通信)
    → 递归到所有叶子节点

  特点: 分散竞争到多个节点，Low Contention

模式C: 对等通信 (Dissemination Barrier)

  每轮(round)中，Block_i 向 Block_(i+2^round) 发送信号:

  Round 0: Block_0→Block_1, Block_1→Block_2, ..., (步长=1)
  Round 1: Block_0→Block_2, Block_1→Block_3, ..., (步长=2)
  Round 2: Block_0→Block_4, Block_1→Block_5, ..., (步长=4)
  ...
  共 log₂(N) 轮

  通信通过全局内存中的标志位数组实现:
    flags[block_id][round] = 1  (写入)
    while(flags[partner_id][round] == 0) {}  (轮询读取)

  特点: 无中心节点，完全对称，最低竞争

四、树形结构如何建立
================================================================================

────────────────────────────────────────────────────────────────────────────────
4.1 Tree Barrier 的树构建方式 (自底向上动态构建)
────────────────────────────────────────────────────────────────────────────────

构建算法: _build_tree(num_blocks)

  参数: num_blocks = N (线程块数量), radix = 2 (扇出度)

  步骤1 - 创建叶子层:
    将N个Block平均分配到 ceil(N/radix) 个叶子节点
    每个叶子节点管理 radix 个Block (最后一个可能不满)

    例: 6个Block, radix=2:
    叶子: node_0(B0,B1)  node_1(B2,B3)  node_2(B4,B5)
    每个叶子的 limit = 管理的Block数 = 2

  步骤2 - 逐层向上合并:
    每 radix 个子节点合并为1个父节点
    父节点的 limit = 子节点数量

    第二层: node_3(node_0, node_1)  node_4(node_2)
    根节点: node_5(node_3, node_4)

  最终树结构 (6个Block):

              node_5 (root, limit=2)
             /                      \
       node_3 (limit=2)         node_4 (limit=1)
       /          \                  |
    node_0        node_1          node_2
    (B0,B1)       (B2,B3)        (B4,B5)
    limit=2       limit=2        limit=2

────────────────────────────────────────────────────────────────────────────────
4.2 Static Tree Barrier 的树构建方式 (完全二叉树数组映射)
────────────────────────────────────────────────────────────────────────────────

构建算法: _build_static_tree(num_blocks)

  采用堆(Heap)的数组映射方式，在编译期/初始化期完全确定树结构。

  参数: num_blocks = N

  步骤1 - 确定总节点数:
    total_nodes = 2*N - 1  (完全二叉树性质)

  步骤2 - 建立父子关系(数组下标映射):
    节点i的左子节点  = 2*i + 1
    节点i的右子节点  = 2*i + 2
    节点i的父节点    = (i-1) // 2
    根节点 = 节点0

  步骤3 - Block映射:
    叶子节点范围 = [N-1, 2N-2]
    Block_i 映射到 节点(N-1+i)

  例: 4个Block:
    total_nodes = 7
    节点编号: 0, 1, 2, 3, 4, 5, 6

    树结构:
                0 (root)
              /         \
            1             2
          /   \         /   \
        3      4      5      6
        B0     B1     B2     B3

    父子映射表:
    ┌────────┬────────┬────────┬────────┬───────────┐
    │ 节点ID │ 父节点 │ 左子   │ 右子   │ Block映射 │
    ├────────┼────────┼────────┼────────┼───────────┤
    │   0    │  None  │   1    │   2    │   -       │
    │   1    │   0    │   3    │   4    │   -       │
    │   2    │   0    │   5    │   6    │   -       │
    │   3    │   1    │  None  │  None  │  Block 0  │
    │   4    │   1    │  None  │  None  │  Block 1  │
    │   5    │   2    │  None  │  None  │  Block 2  │
    │   6    │   2    │  None  │  None  │  Block 3  │
    └────────┴────────┴────────┴────────┴───────────┘

  与Tree Barrier的区别:
    - Static Tree: 节点数固定(2N-1)，结构在初始化时完全确定，不可动态调整
    - Tree Barrier: 根据radix动态构建，节点数随radix变化
    - Static Tree的优势: 父子关系通过简单数学公式计算，无需存储指针
    - 在真实GPU中，这种数组映射方式对全局内存访问更友好(连续内存,更好的cache利用率)

────────────────────────────────────────────────────────────────────────────────
4.3 树形栅栏的到达与释放的硬件过程
────────────────────────────────────────────────────────────────────────────────

到达阶段 (Arrive Phase) —— 自底向上聚合:

  Block到达 → 对所属叶子节点的counter做atomicAdd
  → 如果叶子满了(count==limit)，重置count并向父节点的counter做atomicAdd
  → 逐层向上传播
  → 到达根节点且根节点满了 → 触发释放

  每"向上一层"就是一次全局内存的原子操作 = 1次通信开销
  总通信轮数 = 树的高度 = O(log₂N)

释放阶段 (Release Phase) —— 自顶向下广播:

  根节点翻转sense → 遍历子节点写入新sense → 递归到所有叶子

  每"向下一层写入子节点" = 1次全局内存写操作 = 1次通信开销
  总通信次数 = 内部节点数 = O(N)

  等待中的Block通过轮询叶子节点的sense来检测释放:
    while(node.sense == local_sense) {}  // 每次循环1次全局内存读取
