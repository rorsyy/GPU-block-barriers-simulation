1. 相关技术介绍

本仿真系统致力于通过软件模拟的方式，将 GPU 硬件层面的隐式同步机制具象化。在开始使用前，了解以下底层硬件与软件算法模型有助于更好地理解仿真内容。

1.1 GPU 体系结构与线程块 (Thread Block)

现代 GPU 采用单指令多线程 (SIMT) 架构。计算任务被划分为多个网格 (Grid)，网格内部又划分为多个线程块 (Thread Block)。这些线程块会被调度到 GPU 的流式多处理器 (Streaming Multiprocessor, SM) 上执行。由于硬件资源的限制和各个线程块遇到内存延迟的不同，它们在实际执行时的进度是高度不平衡的。

1.2 什么是栅栏同步 (Barrier Synchronization)

当不同的线程块需要协同完成一项工作时（例如：规约求和、矩阵分块相乘），必须确保所有相关的线程块都完成了当前阶段的任务，才能进入下一阶段。这种强制所有参与者在某一点汇合的机制称为**栅栏 (Barrier)**。

  • 到达 (Arrive)：某个线程块完成了阶段任务，向全局注册自己的状态。
  • 等待/释放 (Wait/Release)：早到的线程块必须在此挂起等待；当最后一个线程块到达时，栅栏状态翻转，唤醒（释放）所有等待中的线程块。

1.3 全局内存与原子操作 (Atomic Instructions)

💡 内存是被动介质 在真实 GPU 中，全局内存 (Global Memory) 只是被动的 DRAM 存储。所有的状态判断和累加操作实际上是由 SM 发起**原子指令**（如 atomicAdd，atomicExch）完成的。这些指令确保了"读-改-写"这一动作在并发执行下绝对安全，但过度使用会造成总线竞争与性能瓶颈。

1.4 核心栅栏算法对比

系统内置了三种经典的同步算法，体现了并行计算中"通信拓扑"的演进：

🟢 集中式栅栏 (Centralized)

原理：所有线程块直接对全局内存中的唯一一个 counter 变量执行原子累加。当 counter 达到阈值时，翻转 sense 标志位进行全局广播。

特点：实现最简单，但 N 个块同时抢夺一个内存地址会导致极严重的锁竞争，通信开销达到 O(N)，大规模时性能极差。

🟡 树形栅栏 (Tree Barrier)

原理：为了解决单点拥堵问题，将线程块分组绑定到叶子节点，到达信号在内部节点中**自底向上逐层汇聚**；当根节点收集完毕后，释放信号再**自顶向下广播**。

特点：将竞争分散到多个局部节点，总线压力急剧降低，通信复杂度优化至 O(log N)。

🔵 静态树形栅栏 (Static Tree Barrier)

原理：在树形结构的基础上，放弃了基于对象的链式引用，改用完全二叉树的**数组索引公式**（如子节点为 `2i+1`, `2i+2`）映射存储节点。

特点：不仅通信复杂度是 O(log N)，其紧凑的内存布局还获得了极佳的缓存局部性，是硬件原生实现的最佳选择。

2. 前端展示的各个要素含义

前端界面由多个高度实时联动的可视化面板组成，用以全面展示仿真运行时的深层状态。

2.1 顶部状态栏

  • 时钟周期 (Tick)：仿真引擎推进的虚拟时间刻度。
  • 栅栏类型：当前运行的同步算法（CENTRALIZED / TREE / STATIC_TREE）。
  • 仿真状态：系统的运行阶段（STOPPED / RUNNING / PAUSED）。
  • 连接状态：前端与后台 Python 引擎的 WebSocket/HTTP 心跳状态。

2.2 线程块状态监控 (Block Status)

状态标签 含义解释
🟢 运行中 线程块正在执行其工作负载，工作量进度条会不断增加。
🟡 等待中 该线程块已经完成了当前阶段任务，发起原子指令并到达了栅栏，目前正在挂起等待其他慢速块。
🔴 已失败 (仅失联模式) 系统随机模拟的宕机或失联故障，该块将不再推进任何工作。
⬜ 已完成 所有的仿真生命周期结束。

2.3 树形同步拓扑 (Topology Animation)

该面板专为树形算法设计，展示层级合并结构和实时信息流传路径：

🌟 橙色脉冲 (通信上传)

当某个线程块到达时，从该叶子节点到其父节点的连线会发出橙色高亮。这不仅代表状态传递，更代表着底层正发生一次 atomicAdd 通信操作。

✅ 绿色渐隐 (全局释放)

当根节点收齐所有子节点的到达信号后，释放信号（sense 翻转）沿全树自顶向下传播，此时全树的连线会闪烁绿色并逐渐隐去，所有等待的线程块被唤醒。

2.4 全局内存存储结构 (Global Memory)

  • 全局内存热力图 (侧边栏)：展示被动存储介质中各变量（如 `node.count`, `sense`）的值。颜色越深（橙色/红色），说明近期发生原子指令交互的频次越高（即热点拥堵区域）。
  • 底层结构代码视图：展示不同的数据结构特征。树形栅栏会渲染为带有指针的邻接表 `node -> [children]`；而静态树形栅栏会渲染为通过公式计算的连续数组结构 `memory[0], memory[1]...`。

2.5 性能指标 (Metrics)

  • 同步次数：整个系统成功完成所有线程块会合释放的累计轮数。
  • 平均通信开销：评判算法优劣的**核心指标**。它统计了为了完成一次同步，SM 到底向全局内存发起了多少次原子操作通信。由于通信代价极大，这个数值越小，算法在 GPU 硬件上的执行效率越高。

3. 使用说明

3.1 仿真参数配置流程

在界面左侧区域，您可以对仿真环境进行参数配置。每次修改后，必须点击 `🔄 重置并应用` 按钮，才能将新配置下发至后台仿真引擎，完成环境重建。

  1. 设置线程块数目 (2-64)：设定参与协作的并行单元规模。
  2. 设置栅栏间隔 (Ticks)：设定两次同步检查点之间的虚拟时钟间隔。
  3. 调节工作量方差 (0.0-1.0)：调节此滑块可控制各线程块工作强度的差异。方差越大，快块与慢块的到达时间差距越明显（模拟重度负载不均衡场景）。
  4. 选择栅栏类型：在集中式、树形、静态树中切换。
  5. 选择仿真模式：选择正常模式，或切换至失联模式模拟故障死锁场景。

3.2 仿真运行与控制

操作 说明
▶️ 启动 启动时钟引擎，系统将自动快速推进（后台步进约50ms/tick）。
⏸️ 暂停 冻结当前帧，此时可以从容观察当前所有线程块的状态及全局内存的瞬时数值。
⏭️ 单步 精细化观察。每次点击，系统仅推进一个时钟周期，非常适合用来捕捉通信发生瞬间的拓扑动画及内存热点。

3.3 典型实验设计：算法开销对比

为加深对不同算法的理解，建议按以下步骤进行一次对比实验：

  1. 将**线程块数目**设为较大的值（如 16 或 32），选择 集中式 (Centralized)，点击重置应用。
  2. 启动仿真，等待发生 2~3 次成功同步后，点击暂停。
  3. 记录此时面板上的 平均通信开销。
  4. 切换为 树形 (Tree)，不修改其他参数，点击重置应用并启动相同周期。
  5. 再次暂停，对比两者的通信开销，您将直观体会到从 O(N) 到 O(log N) 所带来的巨大通信节约。

3.4 键盘快捷键

为提高操作效率，系统支持以下全局快捷键（前提是鼠标焦点不在任何输入框内）：

按键 功能
S 启动仿真 (Start)
P 暂停仿真 (Pause)
Space 单步执行 (Step)
R 重置仿真 (Reset)
⚠️ 故障模式排查 (Troubleshooting) 如果在运行中发现仿真长时间停滞不前（时钟仍在走，但无任何同步完成），请检查是否开启了**失联模式**。在该模式下如果某个线程块"宕机"变为红色，而系统配置的却是集中式栅栏算法，它将不可避免地导致死锁现象。
